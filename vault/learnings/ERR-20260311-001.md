# heartbeat-runner.py overwrites script state files

## What happened
github-watch.py was reporting the same pushes every 15-minute poll cycle. The script correctly saved updated SHAs to its state file, but they kept reverting to old values.

## What was learned
`heartbeat-runner.py` loads `state = load_state(task_name)` BEFORE running the script, then calls `save_state(task_name, state)` AFTER — overwriting whatever the script saved with the pre-run copy plus runner metadata (`last_run`, `total_runs`, etc.). This is a classic read-modify-write race: runner reads old state, script writes new state, runner overwrites with old state.

## Why it matters
Any heartbeat script that persists its own state (SHAs, IDs, timestamps) will have that state silently clobbered every run. This made github-watch send duplicate Discord notifications for every commit on every poll cycle — ~50 runs of duplicates before caught.

## Evidence
- `heartbeat-tasks/heartbeat-runner.py` line 236-253 (before fix)
- `heartbeat-tasks/github-watch.state.json` — SHAs kept reverting to old values
- Fix: re-load state AFTER script execution before merging runner metadata (line 237)

## Fix
Added `state = load_state(task_name)` right after script/claude execution completes, before the runner merges its own metadata. This picks up whatever the script saved, then layers runner metadata on top.
